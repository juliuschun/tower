## 이 문서의 목적

Proactive Agent를 "만들었다"에서 "잘 쓴다"로 넘어가기 위한 문서.
세 가지 걱정을 다룬다:

1. **사용처가 너무 많다** — 어떤 걸 켜고 어떤 걸 끌지, 우선순위
2. **세션이 쌓인다** — 사이드바에서 어떻게 보이고, 어떻게 찾나
3. **관리가 안 된다** — 누가 뭘 쏘고 있는지, 비용은 얼마나 드는지

## Part 1: 사용처 정리 — "뭘 켜야 하나"

### Tier 분류

사용처를 영향도와 빈도로 나눈다. Tier 1만 먼저 켜고, 나머지는 필요할 때 추가.

```
              빈도 높음
                ↑
   Tier 2      │      Tier 1
  (나중에)      │     (먼저 켜기)
                │
 ───────────────┼──────────────→ 영향도 높음
                │
   Tier 4      │      Tier 3
  (안 해도 됨)  │     (가치 있지만 드묾)
                │
              빈도 낮음
```

### Tier 1 — 당장 가치 있고, 자주 쓸 것

| 사용처 | 트리거 | 프롬프트 핵심 | 기대 효과 |
|--------|--------|-------------|----------|
| **모닝 브리핑** | cron: 평일 오전 9시 | 어제 완료된 태스크, 오늘 예정된 것, 미해결 이슈 정리 | 하루 시작을 AI와 함께 |
| **태스크 완료 후속** | task_event: done | 결과 요약 + "후속 작업 필요?" | 태스크 완료를 놓치지 않음 |
| **진화 알림** | heartbeat: delta > 10줄 | progress.md 변화 요약 + 다음 단계 제안 | 프로젝트 상태를 능동적으로 파악 |

### Tier 2 — 가치 있지만, 안정화 후 추가

| 사용처 | 트리거 | 비고 |
|--------|--------|------|
| 주간 회고 | cron: 금요일 17시 | 이번 주 작업 요약 + 다음 주 계획 |
| 코드 리뷰 알림 | GitHub webhook | PR이 오래 열려 있으면 리마인드 |
| 비용 리포트 | cron: 월요일 9시 | 지난 주 토큰 사용량 + 추이 |
| 아이디어 제안 | cron: 수요일 14시 | 최근 작업 패턴 기반 개선 제안 |

### Tier 3 — 특수 상황

| 사용처 | 트리거 | 비고 |
|--------|--------|------|
| 장애 알림 | 모니터링 webhook | 서버 다운, 에러율 급증 |
| 신규 멤버 온보딩 | user_event: created | 프로젝트 소개 + 시작 가이드 |
| 마일스톤 축하 | manual | 프로젝트 완료, 100번째 커밋 등 |

### 관리 원칙: 기본은 OFF

> **Opt-in, not opt-out.**
> 모든 proactive 템플릿은 기본 비활성. 사용자가 명시적으로 켜야 동작한다.
> "알림 지옥"은 이 기능의 가장 큰 적이다.

### 일일 한도

| 구분 | 기본값 | 변경 가능 |
|------|--------|----------|
| 사용자당 일일 proactive 세션 | 최대 5회 | 설정에서 조정 |
| 단일 템플릿 최소 간격 | 4시간 | 템플릿별 설정 |
| Quiet Hours (DND) | 22:00–08:00 KST | 사용자별 설정 |

## Part 2: 세션이 쌓이는 문제 — "어디서 어떻게 찾나"

이게 핵심 걱정이다. Proactive 세션이 하루 3~5개씩 쌓이면
일반 세션과 섞여서 사이드바가 엉망이 된다.

### 현재 세션 UI 구조

```
Sidebar (Sessions 탭)
├── Project A
│   ├── label: "feature" ── 세션들
│   └── label: "bug" ── 세션들
├── Project B
│   └── ...
└── (프로젝트 없는 세션들)
```

세션에는 이미 `label` 필드가 있고, 사이드바는 label별로 그룹핑할 수 있다.
또한 `[task]` 접두사 세션은 "task" 뱃지, `channel_ai` 라벨은 "agent" 뱃지로
시각적으로 구분하는 패턴이 이미 있다.

### 해결 전략: 3단계

#### 단계 1 — 지금 바로 (코드 변경 최소)

**Proactive 세션에 시각적 구분자 추가:**

`SessionItem.tsx`에서 `label === 'proactive'`일 때 뱃지 표시:

```typescript
// 기존 패턴:
// isKanbanTask → "task" 뱃지 (amber)
// isChannelAi → "agent" 뱃지 (cyan)
// 추가:
// label === 'proactive' → "💬" 뱃지 (blue)
```

**효과**: 세션 목록에서 proactive 세션이 눈에 띈다. 무엇인지 바로 안다.

#### 단계 2 — Deck 필터 활용

Sidebar의 "Deck" 필터(`filterLabels`)가 이미 label별 그룹핑을 지원한다.
`proactive` 라벨이 자동으로 하나의 덱이 된다.

```
Sidebar (Deck 필터 ON)
├── 📂 proactive ── (AI가 건 대화들)
│   ├── 💬 모닝 브리핑 (오늘)
│   ├── 💬 태스크 후속 (1시간 전)
│   └── 💬 진화 알림 (어제)
├── 📂 feature ── (일반 작업들)
└── 📂 bug
```

**추가 구현 필요 없음** — label 기반 그룹핑은 이미 동작한다.
사용자가 Deck 필터를 켜면 proactive 세션이 자연스럽게 분리된다.

#### 단계 3 — Proactive 전용 뷰 (Phase 2)

Settings 또는 NotificationBell 안에 "Proactive" 탭:

```
┌─ Proactive ─────────────────────────────────┐
│                                              │
│  오늘                                        │
│  ├─ 💬 모닝 브리핑         9:03 AM   열기 →  │
│  │   "어제 3개 태스크 완료..."               │
│  ├─ 💬 태스크 후속         2:15 PM   열기 →  │
│  │   "배포 파이프라인 결과..."               │
│  │                                           │
│  어제                                        │
│  ├─ 💬 진화 알림          11:00 AM   열기 →  │
│  │   "KAP 프로젝트에 12줄..."               │
│  └─ 💬 모닝 브리핑         9:01 AM           │
│      "어제 2개 태스크..."    (읽음, 답장함)  │
│                                              │
│  이번 주 통계: 8회 실행, 5회 열람, 3회 답장   │
└──────────────────────────────────────────────┘
```

### 자동 정리 (TTL)

Proactive 세션은 대부분 일회성이다.
사용자가 답장하지 않은 proactive 세션은 자동으로 아카이브:

| 상태 | TTL | 처리 |
|------|-----|------|
| 미열람 (delivered) | 7일 | 자동 아카이브 |
| 열람했지만 미답장 (opened) | 14일 | 자동 아카이브 |
| 답장함 (replied) | 일반 세션과 동일 | 수동 관리 |

이렇게 하면 사이드바에 proactive 세션이 무한히 쌓이지 않는다.

## Part 3: 관리 — "누가 뭘 쏘고 있는지"

### 관리 대시보드 (Phase 2)

Admin Settings에 "Proactive Agent" 섹션:

```
┌─ Proactive Agent 관리 ──────────────────────┐
│                                              │
│  활성 템플릿                                  │
│  ┌──────────────────────────────────────────┐│
│  │ ☑ 모닝 브리핑    평일 9시   sonnet  ✏️  ││
│  │ ☑ 태스크 후속    이벤트     sonnet  ✏️  ││
│  │ ☐ 주간 회고      금 17시    sonnet  ✏️  ││
│  │ ☐ 비용 리포트    월 9시     haiku   ✏️  ││
│  │                          [+ 새 템플릿]   ││
│  └──────────────────────────────────────────┘│
│                                              │
│  이번 주 통계                                │
│  ├─ 실행: 12회                               │
│  ├─ 열람률: 83% (10/12)                      │
│  ├─ 답장률: 42% (5/12)                       │
│  └─ 예상 토큰: ~24K (약 $0.07)               │
│                                              │
│  설정                                        │
│  ├─ 일일 한도: [5] 회                        │
│  ├─ Quiet Hours: [22:00] ~ [08:00]           │
│  └─ 외부 알림: ☑ Telegram  ☐ Kakao          │
└──────────────────────────────────────────────┘
```

### 비용 통제

Proactive는 서버가 자동으로 Claude를 실행하는 것이므로 비용이 예측 가능해야 한다.

| 시나리오 | 예상 토큰/회 | 일 5회 기준 | 월간 |
|----------|-------------|------------|------|
| 짧은 인사 (sonnet) | ~2K in + ~200 out | ~11K | ~330K |
| 브리핑 (sonnet) | ~5K in + ~500 out | ~27K | ~825K |
| 분석 (sonnet) | ~10K in + ~1K out | ~55K | ~1.65M |

Claude Max 구독이면 토큰 비용은 없지만, 일일 한도에 영향.
API 과금이면 월 ~$2~5 수준 (Tier 1만 사용 시).

### 로그와 추적

`proactive_logs` 테이블 (Phase 2)로 모든 실행을 추적:

```
fired_at    | template     | user  | status    | opened_at | replied_at
────────────┼──────────────┼───────┼───────────┼───────────┼───────────
04-12 09:03 | 모닝 브리핑   | admin | replied   | 09:05     | 09:07
04-12 14:15 | 태스크 후속   | admin | opened    | 14:20     | null
04-11 09:01 | 모닝 브리핑   | admin | delivered | null      | null
```

열람률/답장률이 낮은 템플릿은 자동으로 비활성 제안.

## Part 4: UX 플로우 — 사용자 경험 시나리오

### 시나리오 1: 아침 출근

```
1. 사용자가 Tower 접속
2. 🔔 벨에 빨간 뱃지 "1"
3. 클릭 → "💬 모닝 브리핑" 알림
4. 알림 클릭 → 세션 열림
5. Claude: "좋은 아침이에요! 어제 완료된 태스크 정리했습니다..."
6. 사용자: "고마워, 오늘은 배포 파이프라인 먼저 볼게"
7. 일반 대화 계속
```

### 시나리오 2: 태스크 완료 후

```
1. 백그라운드 태스크가 완료됨
2. 기존: 🔔 "✅ 태스크 완료" (읽고 끝)
3. Proactive: 🔔 "💬 태스크 후속" (새 세션)
4. Claude: "방금 배포 파이프라인 태스크가 완료됐어요.
           결과를 보니 staging 환경에서 2개 경고가 있네요.
           확인해볼까요?"
5. 사용자: "경고 내용 보여줘"
6. 대화로 이어짐
```

### 시나리오 3: 알림 과다 (안 좋은 경우)

```
1. 사용자가 Tower 접속
2. 🔔 벨에 빨간 뱃지 "7"
3. 전부 proactive... 모닝 브리핑, 진화 알림 3개, 태스크 후속 2개, 주간 회고
4. 사용자: "아 시끄러워" → 전체 무시
5. 열람률 하락 → 기능 꺼버림
```

**이걸 막는 방법:**
- 일일 한도 (기본 5회)
- Quiet Hours
- 답장률 낮으면 자동 빈도 감소 제안
- 알림 그룹핑: 같은 날 여러 proactive를 하나의 알림으로 묶기

## Part 5: 기존 세션 주입 — 언제 쓰나

새 세션 생성이 기본이지만, 기존 세션에 주입이 더 자연스러운 경우:

| 상황 | 모드 | 이유 |
|------|------|------|
| 브리핑, 정기 알림 | **새 세션** | 독립적 맥락, 깔끔한 시작 |
| 태스크 완료 후속 | **새 세션** | 태스크와 별개의 후속 대화 |
| 진행 중인 작업 리마인드 | **기존 세션 주입** | "아까 그 작업 이어서..." |
| 장시간 대화 후 팔로업 | **기존 세션 주입** | 맥락 유지가 핵심 |
| 에러 발생 후 디버깅 제안 | **기존 세션 주입** | 에러 컨텍스트 필요 |

기존 세션 주입 시 주의:
- 세션이 streaming 중이면 스킵 (충돌 방지)
- 주입된 메시지에 시각적 구분자 (시스템 메시지 스타일)
- resume 실패 시 새 세션으로 폴백

## 핵심 원칙 요약

1. **기본은 OFF** — Opt-in. 사용자가 명시적으로 켜야 동작.
2. **적게, 의미 있게** — 하루 5회 이내. 빈 인사 금지. 맥락 필수.
3. **찾기 쉽게** — proactive 뱃지 + Deck 필터 + 전용 뷰.
4. **쌓이지 않게** — 미열람 7일, 미답장 14일 후 자동 아카이브.
5. **끄기 쉽게** — 템플릿별 토글. 전체 OFF 원클릭.
6. **비용 예측 가능** — 일일 한도 + 모델 선택 + 월간 리포트.
